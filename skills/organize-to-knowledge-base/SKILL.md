# 整理到知识库（organize-to-knowledge-base）

把 AI 生成的文档 / 结论 / 方案 / 调研笔记等内容，**分类**归档进本地知识库，然后**只提交本次归档文件**并推送到远端。

> **与知识网体系的关系**：本技能负责把「整篇成品」放进分类目录（技术分享/临时想法/…）并做安全提交推送，是底层落盘+提交原语。若你要的是「边学边沉淀、原子笔记 + 双链 + 编入知识网」，用 `kb-digest`（消化入库）/ `kb-weave`（编织）/ `kb-ask`（问答）三件套；它们复用本技能 `scripts/` 下的 `commit_kb.sh`（安全提交）与 `kb_search.sh`（结构化检索）。整库重活派 `kb-librarian` agent。约定见知识库根目录 `README.md`。

## 知识库位置

- 知识库根目录：`/Users/liusiyang/Documents/GitHubOfMine/AboutAI/知识库`
- 它是 `AboutAI` 这个 git 仓库（远端 `github.com/liusiyangiOS/AboutAI`）下的**子目录**，不是独立仓库。提交推送针对整个仓库，但**只暂存本次归档的文件**。

## 整体流程

按顺序执行四步。每一步做完再进入下一步。

### 第 1 步：采集内容

内容可能来自三种来源，按用户当前语境判断：

- **当前对话上下文**：把本次对话里 AI 生成的文档 / 结论整理出来（最常见）。
- **指定的文件 / 目录**：用户给了路径，用 Read 读取；目录则整理其中相关文件。
- **手动粘贴的文本**：用户在对话里贴了一段内容，直接归档。

采集时保留原始信息，不要臆造。如果内容很零散，先在脑子里理成一篇结构清晰的 markdown 再落盘。

### 第 2 步：分类（预设分类 + 动态补充）

先对照下面的**预设分类表**给内容归类；只有当没有任何一类合适时，才新建一个语义清晰的目录。

| 目录 | 适合放什么 |
|------|-----------|
| `技术分享/` | 技术文章、原理讲解、可对外分享的知识沉淀 |
| `工具与方法/` | AI 工具用法、工作流、最佳实践（如 Claude Code / Codex 用法、prompt 技巧） |
| `调研笔记/` | 技术调研、竞品分析、开源项目评估的结论 |
| `方案设计/` | 架构 / 实现方案、设计决策与取舍 |
| `复盘总结/` | 项目复盘、经验教训、事故回顾 |
| `临时想法/` | 未成型的灵感、待整理的碎片（收件箱性质） |

规则：

- **落盘前，把最终选定的目录名（含"是复用现有目录还是新建目录"）明确告诉用户**，再写文件。这一步很轻，一句话说明即可，不必等用户回复——除非要**新建**一个预设表之外的目录，那就先确认目录名，避免知识库长出一堆命名不一致的目录。
- 现有目录里可能已有更细的层级（如 `技术分享/刘昭/`）。若内容明显属于某个已存在的子目录，优先沿用，保持结构一致。
- 一次归档多篇、且分属不同类时，分别放进各自目录。

### 第 3 步：写入文件

- 文件格式：**markdown（.md）**，正文用中文（遵循用户全局偏好）。
- 命名：`YYYY-MM-DD-简短标题.md`，日期用 `date +%F` 取当天，标题用简短中文短语（空格用 `-` 连接，不要特殊字符）。日期前缀便于按时间排序。
- 每个文件顶部加一段 frontmatter 元信息，便于日后检索：

```markdown
---
title: 子代理编排方法
category: 技术分享
created: 2026-07-05
source: 对话上下文        # 或：文件 <路径> / 手动粘贴
tags: [Claude Code, 多代理, 工作流]
---

# 子代理编排方法

<正文……>
```

- 正文用清晰的标题层级组织；如果是"结论"类短内容，也要有个一级标题和简短背景，别只丢一句话进去。

### 第 4 步：提交并推送（只提交本次归档文件）

**必须使用** `scripts/commit_kb.sh`，不要手动 `git add -A` / `git add .`——仓库里可能有用户遗留的无关改动，全量提交会误伤。

```bash
bash ${CLAUDE_PLUGIN_ROOT}/skills/organize-to-knowledge-base/scripts/commit_kb.sh "<commit message>" <本次写入的文件1> [文件2 ...]
```

- commit message 用中文、说明归档了什么到哪个分类，例如：
  `docs(知识库): 归档 Claude Code 子代理编排方法到 技术分享`
- 脚本会：只暂存传入的文件 → 打印本次将提交的改动 stat → 提交 → 推送到当前分支 upstream（无 upstream 则 `push -u origin <branch>`）→ 打印最新提交。
- **展示后自动推送**（用户已确认此默认行为）：脚本执行完，把 commit 摘要和推送结果简要汇报给用户即可，无需在 push 前再停下确认。

## 收尾汇报

归档完成后，用中文简要告诉用户：

- 归档了哪些内容、放进了哪个 / 哪些目录（给出相对路径）；
- 是否新建了目录；
- commit message 与推送结果（成功 / 失败原因）。

## 边界与安全约束

- **只动本次归档文件**：绝不把仓库里其它未提交改动纳入本次提交。若脚本检测到没有可提交内容，会跳过提交与推送并说明。
- **推送失败**（如远端有新提交、鉴权失败）：不要强推，把错误原样反馈给用户，建议先 `git pull --rebase` 或让用户处理，再重试。
- **新建目录要克制**：能归入预设分类就不新建；确需新建先跟用户确认目录名。
- **不臆造内容**：归档的是已有的 AI 产物或用户给的材料，不要自行扩写结论。
