# kb-weave — 把新知识编进知识网

知识的价值在连接，不在堆叠。本技能保证每条新知识入库时都**长进已有的网**，而不是躺成孤岛。这是复利的来源。

## 一句话：kb-weave 到底在干什么

你新写了一条笔记，它此刻是**孤岛**——库里可能已有三条和它相关的旧笔记，但彼此不知道对方存在。kb-weave 做四件事，把这座孤岛接进大陆：

1. **找亲戚**：在库里搜出和新笔记相关的旧条目。
2. **牵线（双向链接）**：让新笔记和旧条目互相指向对方——两边都写上 `[[对方]]`，缺一边就是坏链。
3. **上地图（更新 MOC）**：把新笔记登记到它所属领域的目录页（MOC）上，让你以后从领域入口就能找到它。
4. **回头修旧账**：如果新知识**推翻或补充**了某条旧笔记，回去把旧笔记也更新一下，并留下"据新笔记修正"的痕迹。

做完这些，你的知识就从"一堆孤立文件"变成"一张越织越密的网"——这正是"不做 RAG 也能又快又准"的物理基础：问答时顺着链接和 MOC 就能定位，不需要向量检索。

## 名词速查

- **原子笔记**：`notes/` 下一条只讲一个概念的小笔记（由 kb-digest 炼成）。
- **双向链接**：两条笔记 frontmatter 的 `related` 里互相写 `[[notes/对方文件名]]`。
- **MOC（Map of Content，领域地图）**：`_MOC/` 下的领域目录页，如 `_MOC/AI工具.md`，列出该领域有哪些条目、哪些开放问题。它是"活的目录"，每次入库都长一点。

## 何时用

- 有新原子笔记入库后（常由 `kb-digest` 或入库流程接力调用）。
- 用户说"编进知识网""关联旧条目""更新 MOC/索引""回头把相关的旧条目更新一遍"。

## 前置：确认知识库位置（首次会问一次，之后不再问）

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/resolve_kb.sh"
```

- 打印出一行路径 → 作为知识库根（记为 `<KB>`）。
- 退出码 3（未配置）→ **只问一次**："请告诉我你的知识库根目录绝对路径（例如 `~/Documents/知识库`）"，然后：

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/set_kb.sh" "<用户给的绝对路径>"
```

格式与目录约定以 `<KB>/README.md` 为准。

## 流程

### 1. 找相关旧条目

用共享检索原语，按新笔记的 `tags` 与标题关键词检索：

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/kb_search.sh" <关键词1> <关键词2> ...
```

读命中靠前的几条（不要全读），判断哪些**真的相关**——宁缺毋滥，别为了连接而连接。

### 2. 建立双向链接（必须成对）

- 在新笔记 frontmatter 的 `related` 里加 `[[notes/旧条目]]`。
- **同时**在那条旧笔记的 `related` 里补回 `[[notes/新笔记]]`。缺一边就是坏链。

### 3. 更新领域 MOC

- 新笔记 `up` 指向的领域 MOC（如 `_MOC/AI工具.md`），在其「条目」区加一行：`- [[notes/新笔记]] — 一句话摘要`。
- 若属于全新领域，在 `_MOC/index.md` 加该领域，并新建 `_MOC/<领域>.md`。

### 4. 回改旧条目（复利的关键动作）

若新知识**修正或补充**了某条旧条目：

- 补充：在旧条目相关位置加一句"另见 [[notes/新笔记]]：补充了 X"。
- 修正：更新旧条目的结论，并在「我的疑问与发散」或正文注明"据 [[notes/新笔记]] 修正：原先认为…，现在…"。

改动要**保守、可追溯**——只动确有必要的地方，保留演进痕迹，别抹掉历史判断。

### 5. 提交推送

把本次**所有被动过的文件**（新笔记 + 被补链/回改的旧笔记 + MOC）用底层原语提交，只提交这些：

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/commit_kb.sh" "docs(知识库): 编织 <主题>，关联 N 条并更新 MOC" <file1> <file2> ...
```

若本技能是被 `kb-digest` 链式调用的，则**由上层统一提交**，此处不重复提交——避免碎提交。

## 原则

- **双向链接成对**，坏链会毒化知识网。
- **MOC 是活的**：每次入库都让它长一点。
- **回改保守**：修正旧条目要留演进痕迹，不做无痕覆盖。
- **只提交动过的文件**，绝不 `git add -A`。
