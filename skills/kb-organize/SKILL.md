# kb-organize — 整篇成品快速归档

把一篇**已经成型、不需要再消化学习**的文档 / 结论 / 方案 / 调研笔记，按类别归档进知识库的「成品区」，然后**只提交本次文件**并推送。

## 这条技能 vs kb-digest（先分清，别用错）

知识库里有两类内容，对应两条不同的流：

| | kb-organize（本技能） | kb-digest |
|---|---|---|
| 面向 | 已写好的**整篇成品** | 外部材料（链接/文章/视频/仓库） |
| 动作 | 分类落盘、存档 | 先讲解让你读懂 → 承接你的追问发散 → 炼成原子笔记 |
| 产物 | 分类目录下的整篇 md（`技术分享/…`） | `notes/` 下的**原子笔记**（含"我的疑问与发散"）|
| 何时 | "这篇写好了，收进去" | "这个讲的啥，我想学明白再存" |

一句话：**已经懂了、只想存档 → 本技能；还想学、要沉淀思考 → kb-digest。**

## 前置：确认知识库位置（首次会问一次，之后不再问）

先解析知识库根目录：

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/resolve_kb.sh"
```

- 打印出一行路径 → 用它作为知识库根（下文记为 `<KB>`）。
- 退出码为 3（未配置）→ **只问用户一次**："请告诉我你的知识库根目录绝对路径（例如 `~/Documents/知识库`）"。拿到后记录，之后自动复用：

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/set_kb.sh" "<用户给的绝对路径>"
```

拿到 `<KB>` 后，落盘前先读 `<KB>/README.md`（知识库的格式与目录约定，唯一事实源）。

## 流程

### 第 1 步：采集内容

三种来源，按当前语境判断：

- **当前对话上下文**：把本次对话里 AI 生成的文档 / 结论整理成一篇（最常见）。
- **指定的文件 / 目录**：用户给了路径，用 Read 读取。
- **手动粘贴的文本**：用户贴了内容，直接归档。

保留原始信息，不臆造。内容零散时先理成一篇结构清晰的 markdown 再落盘。

### 第 2 步：分类（预设分类 + 动态补充）

先对照预设分类表归类；只有当没有任何一类合适时，才新建一个语义清晰的目录。

| 目录 | 适合放什么 |
|------|-----------|
| `技术分享/` | 技术文章、原理讲解、可对外分享的知识沉淀 |
| `工具与方法/` | AI 工具用法、工作流、最佳实践（如 Claude Code / Codex 用法、prompt 技巧） |
| `调研笔记/` | 技术调研、竞品分析、开源项目评估的结论 |
| `方案设计/` | 架构 / 实现方案、设计决策与取舍 |
| `复盘总结/` | 项目复盘、经验教训、事故回顾 |
| `临时想法/` | 未成型的灵感、待整理的碎片（收件箱性质） |

规则：

- **落盘前把选定目录（复用现有 or 新建）一句话告诉用户**再写，不必等回复——除非要**新建**预设表之外的目录，则先确认目录名，避免长出一堆命名不一致的目录。
- 现有目录可能已有更细层级（如 `技术分享/刘昭/`）。内容明显属于某个已存在子目录时，优先沿用。
- 一次归档多篇且分属不同类时，分别放进各自目录。

### 第 3 步：写入文件

- 格式 **markdown（.md）**，正文中文。
- 命名：`YYYY-MM-DD-简短标题.md`，日期用 `date +%F`，标题用简短中文短语（空格用 `-` 连接，无特殊字符）。
- 顶部加 frontmatter：

```markdown
---
title: 子代理编排方法
category: 技术分享
created: 2026-07-05
source: 对话上下文        # 或：文件 <路径> / 手动粘贴
tags: [Claude Code, 多代理, 工作流]
---

# 子代理编排方法

<正文……>
```

- 正文用清晰标题层级；即便是"结论"类短内容，也要有一级标题和简短背景，别只丢一句话。

### 第 4 步：提交并推送（只提交本次归档文件）

**必须使用** `commit_kb.sh`，不要手动 `git add -A` / `git add .`——仓库里可能有用户遗留的无关改动，全量提交会误伤。

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/commit_kb.sh" "<commit message>" <本次写入的文件1> [文件2 ...]
```

- commit message 用中文，说明归档了什么到哪个分类，例如：
  `docs(知识库): 归档 子代理编排方法 到 技术分享`
- 脚本会：只暂存传入文件 → 打印本次改动 stat → 提交 → 推送（无 upstream 则 `push -u origin <branch>`）→ 打印最新提交。
- **展示后自动推送**：脚本跑完把 commit 摘要和推送结果简要汇报即可，无需在 push 前再停下确认。

## 收尾汇报

用中文简要告诉用户：归档了什么、放进哪个目录（相对路径）、是否新建目录、commit message 与推送结果。

## 边界与安全约束

- **只动本次归档文件**：绝不把仓库其它未提交改动纳入提交；无可提交内容时脚本会跳过并说明。
- **推送失败**（远端有新提交 / 鉴权失败）：不强推，把错误原样反馈，建议先 `git pull --rebase` 再重试。
- **新建目录要克制**：能归入预设分类就不新建；确需新建先确认目录名。
- **不臆造内容**：归档的是已有产物或用户给的材料，不自行扩写结论。
