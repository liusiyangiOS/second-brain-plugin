# kb-digest — 先消化，再入库

把外部材料变成"你真正读懂、并且连你的疑问都留下来"的知识。这是第二大脑里最日常的一条流。

格式与目录约定以知识库根目录 `/Users/liusiyang/Documents/GitHubOfMine/AboutAI/知识库/README.md` 为准，落盘前先读它。

## 何时用

- 用户发来链接 / 文章 / 视频 / 开源仓库 / 文件，想先搞懂再决定要不要沉淀。
- 用户说"这个是干啥的""帮我总结""我看看懂不懂"。
- 用户在一轮讲解后说"懂了 / 入库 / 收进去 / 沉淀下来"。

## 流程

分四步，**不要跳过第 2、3 步直接入库**——先读懂是这条流的意义所在。

### 1. 取材

- 链接 → 用 `WebFetch` 抓取正文（视频链接抓取简介/字幕页；仓库抓 README）。
- 文件 / 目录 → 用 `Read` 读。
- 粘贴文本 → 直接用。

抓不到就如实说明，别脑补内容。

### 2. 讲解 + 总结（先不入库）

用中文给出：

- **一句话定性**：这材料到底是干啥的、解决什么问题、适合谁。
- **结构化摘要**：核心观点 / 关键步骤 / 重要结论，分点列清。
- **和你已有知识的关系**（可选）：先用 `kb_search.sh` 扫一下库里有没有相关旧条目，若有就点一句"这和你之前记的 X 有关"。

讲完**停下来**，让用户消化。明确告诉用户：可以继续追问，觉得没问题了说一声就入库。

### 3. 承接追问与发散（关键差异点）

- 用户不懂就解释，想展开就一起发散。
- **边答边记**：把这一轮的关键问答整理成"疑问与发散"草稿（放临时变量/草稿，先不落盘）。这些是复利的种子，别丢。
- 反复第 3 步，直到用户说读懂了 / 要入库。

### 4. 消化后入库 + 编织

用户示意入库后：

1. 按 README 的原子笔记格式，在 `notes/` 建 `YYYY-MM-DD-简短标题.md`：
   - `## TL;DR`：把材料炼成半成品（一到三句）。
   - `## 核心论断`：关键结论。
   - `## 我的疑问与发散`：**填入第 3 步沉淀的问答**——这一节是本技能的灵魂。
   - frontmatter 填 `source`（原链接/文件）、`tags`、`status: 已消化`。
2. **接着走 `kb-weave` 技能**：找相关旧条目、建双向链接、更新对应 MOC（复利引擎）。
3. 用 `${CLAUDE_PLUGIN_ROOT}/skills/organize-to-knowledge-base/scripts/commit_kb.sh` 提交推送**本次涉及的所有文件**（新笔记 + 被更新的旧笔记 + MOC），只提交这些、不 `git add -A`。

## 原则

- **先读懂再沉淀**：没消化就别入库，否则只是搬运。
- **疑问必留**：用户当时不懂的点、追问、展开，和结论一样值钱。
- **不臆造**：沉淀的是材料本身 + 用户的真实思考，不自行编造结论。
