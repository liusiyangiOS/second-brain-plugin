# 第二大脑 · 个人知识库工具（second-brain-plugin）

一个把「AI 生成的内容」沉淀成「可复利的个人知识网」的 Claude Code 插件。

它不做 RAG——不把原料丢进向量库现查，而是**在入库时就把算力前置**：把材料炼成原子笔记、建立双向链接、更新领域地图（MOC）。提问时只需读「地图 + 少量相关笔记」，又快又准，且越用越聪明。

## 设计理念

这套工具想解决的，是"信息越存越多、却越来越不像知识"的问题。它的立场借用一句话概括：

> 「知识是反复加工、彼此连接之后的产物，不是原始信息的堆叠。」
> —— 引自 stormzhang 公众号文章《专属自己的第二大脑——个人知识库》

顺着这个立场，落到四条原则：

- **半成品优先，不做 RAG**：入库即炼成原子笔记（TL;DR + 核心论断 + 你的疑问与发散），提问时只读半成品，不做向量检索。这一点受 Andrej Karpathy 关于"把原料提前炼成半成品、提问时又快又准"的知识库思路启发。
- **入库必编织**：每条新知识都要长进已有的网——找相关旧条目、建双向链接、更新 MOC，必要时回头修订旧条目。这是复利的来源。
- **沉淀疑问，不只沉淀结论**：你当时没懂的点、追问、发散，和结论一样值钱，一并留下。
- **git-native + Claude-native**：知识库就是一堆 markdown + 约定，Claude 本身就是检索与编织引擎，不依赖任何 GUI 应用。

> 关于借鉴：本插件受 stormzhang「第二大脑」与 Karpathy 个人知识库思路启发，但**没有照搬任何一方的方案**——结构、技能划分、脚本都是针对"git 里的 markdown 知识库 + Claude Code 工作流"这一具体场景重新设计的。上面加引号的那句话是 stormzhang 原文，特此标注出处；其余为作者自述。详见文末「致谢与声明」。

## 名词速查（看懂下面的能力先看这里）

- **原子笔记**：`notes/` 目录下一条**只讲一个概念、能独立看懂、并连向相关条目**的小笔记。它由 kb-digest 把一篇材料炼成三段：`## TL;DR`（一句话精华）、`## 核心论断`（关键结论）、`## 我的疑问与发散`（你当时的追问与联想）。
  - 它和「整篇成品」不是一回事：整篇成品是一篇完整文章（放 `技术分享/` 等分类目录，由 kb-organize 归档）；原子笔记是被炼小、便于连接和快速问答的半成品。
- **MOC（Map of Content，领域地图）**：`_MOC/` 下的领域目录页（如 `_MOC/AI工具.md`），列出该领域有哪些条目、哪些开放问题。它是"活的目录"，每次入库都长一点。
- **双向链接**：两条相关笔记在各自 frontmatter 的 `related` 里互相写 `[[notes/对方]]`。缺一边就是坏链。
- **编织（weave）**：把新笔记接进知识网的动作——找亲戚、牵线、上地图、必要时回头修旧账。

## 组成

| 名称 | 类型 | 作用 |
|------|------|------|
| `kb-digest` | skill | 给链接/材料 → 先讲解总结（不入库）→ 承接你的追问发散 → 消化后炼成原子笔记入库 |
| `kb-weave` | skill | 入库即编入知识网：双向链接、更新 MOC、回改旧条目（复利引擎） |
| `kb-ask` | skill | 基于炼好的半成品快速问答，不做 RAG，答案溯源到具体笔记 |
| `kb-organize` | skill | 整篇成品快速归档：分类落盘 + 只提交本次文件后推送 |
| `kb-librarian` | agent | 整库重活：巡检死链、补全双链、合并重复、批量更新 MOC |

底层脚本（`scripts/`）：
- `resolve_kb.sh` / `set_kb.sh`：解析 / 设置知识库位置（见下「首次配置」）。
- `kb_search.sh`：结构化检索（非 RAG）。
- `commit_kb.sh`：只暂存本次文件的安全提交，绝不 `git add -A`。

## 使用流程

四条主线，按你当下的意图选：

1. **边学边沉淀（最常用）** → 说「帮我看看这个链接讲的啥」触发 `kb-digest`：
   它先讲解总结、停下来等你追问；你问到懂了说「入库」，它把材料连同**你的疑问发散**炼成原子笔记，接着自动 `kb-weave` 编进知识网，最后只提交本次文件并推送。
2. **成品直接归档** → 说「把这篇整理到知识库」触发 `kb-organize`：
   你已有一篇写好的文档，不需要再学，按分类落盘存档即可。
3. **问自己的库** → 说「问下我的知识库，我之前怎么理解 X 的」触发 `kb-ask`：
   它翻你已沉淀的笔记作答，并指回是哪条；库里没有就如实说没有。
4. **定期体检** → 说「给知识库做次全面体检」派发 `kb-librarian`：
   巡检死链、补全双链、合并重复、补 MOC，先给报告再动手。

> kb-digest 与 kb-organize 的区别：**还想学、要沉淀思考 → kb-digest；已经懂了、只想存档 → kb-organize。**

## 安装

```
/plugin marketplace add liusiyangiOS/second-brain-plugin
/plugin install second-brain-plugin@second-brain
```

- 第一条把本仓库注册成一个插件市场（marketplace 名 `second-brain`）。
- 第二条从该市场安装插件（插件名 `second-brain-plugin`），所以是 `插件@市场` 的写法。
- 这里用的是 `owner/repo` 简写（GitHub 源），不必写 `https://`——和安装任何公开插件的命令一样。

## 首次配置（解决"别人装完还要改路径 + 不暴露作者本地路径"）

插件**不写死任何知识库路径**。第一次用某个 kb 技能时，它会问你一次知识库根目录在哪，然后记住，之后不再问。你也可以手动设置：

```bash
bash <plugin>/scripts/set_kb.sh /你的/知识库/绝对路径
```

解析优先级：环境变量 `SECOND_BRAIN_KB_PATH` > 配置文件 `${CLAUDE_PLUGIN_DATA:-~/.config/second-brain-plugin}/kb_path`。想临时切换到另一个知识库，`export SECOND_BRAIN_KB_PATH=/另一个/库` 即可覆盖。

> 前提：你的知识库目录本身是一个 git 仓库（`git init` 并设好远端），这样 `commit_kb.sh` 才能提交推送。

## 架构：为什么工具公开、知识库私有是对的

- **本仓库（工具）公开** —— 就像 VS Code 开源。
- **你的知识库（数据）私有** —— 就像你的代码不必开源。

插件通过 `resolve_kb.sh` 在运行时才拿到你的知识库位置，**插件代码里不含任何人的本地路径**，所以公开它不会泄露你的知识库在哪、更不会泄露内容。换新机器时：`git clone` 你的私有知识库仓库拿数据，装本插件拿工具，两者解耦、各自独立。

## 知识库结构约定

```
知识库/
├── _MOC/            # 领域地图（知识网枢纽，随新笔记不断生长）
├── notes/           # 原子笔记：一条一个概念，frontmatter 双链
├── 技术分享/ 临时想法/ …   # 成品 / 归档分类目录
└── README.md        # 约定唯一事实源
```
