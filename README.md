# 第二大脑 · 个人知识库工具（second-brain-plugin）

一个把「AI 生成的内容 / 你读到的材料」沉淀成「可复利的个人知识网」的 Claude Code 插件。

它的立场很明确：**知识是反复加工、彼此连接之后的产物，不是原始信息的堆叠。** 所以这套工具不做 RAG——不把原料丢进向量库现查，而是**在入库时就把算力前置**：把材料炼成原子笔记、建立双向链接、更新领域地图（MOC）。提问时只需读「地图 + 少量相关笔记」，又快又准，且越用越聪明。

> 关于命名：插件叫 **second-brain（第二大脑）**，是这套系统的整体隐喻；里面各技能操作的对象是你的 **knowledge base（知识库）**，所以技能统一用 `kb-` 前缀（`kb` = knowledge base 的中性技术缩写）。"大脑里存放着知识库"——两个名字各司其职，不必同名。

## 设计理念

- **半成品优先，不做 RAG**：入库即炼成原子笔记（TL;DR + 核心论断 + 你的疑问与发散），提问时只读半成品，不做向量检索。
- **入库必编织**：每条新知识都要长进已有的网——找相关旧条目、建双向链接、更新 MOC，必要时回头修订旧条目。这是复利的来源。
- **沉淀疑问，不只沉淀结论**：你当时没懂的点、追问、发散，和结论一样值钱，一并留下。
- **git-native + Claude-native**：知识库就是一堆 markdown + 约定，Claude 本身就是检索与编织引擎，不依赖任何 GUI 应用。

## 核心名词（先搞懂这几个）

| 名词 | 是什么 | 放在哪 |
|------|--------|--------|
| **原子笔记** | 一条**只讲一个概念、能独立看懂、并连向相关条目**的小笔记。由 kb-digest 把一篇材料炼成三段：`## TL;DR`（一句话精华）+ `## 核心论断` + `## 我的疑问与发散`。 | `notes/` |
| **整篇成品** | 一篇完整文章 / 结论 / 方案，信息量大、适合通读。就是你知识库里那种原始的技术文档。由 kb-organize 归档。 | `技术分享/` 等分类目录 |
| **双向链接** | 两条笔记 frontmatter 的 `related` 里**互相**写 `[[对方]]`。缺一边就是坏链。 | 笔记 frontmatter |
| **MOC（领域地图）** | Map of Content，一个领域的"活目录页"，列出该领域有哪些条目、哪些开放问题。每次入库都长一点。 | `_MOC/` |

> **原子笔记 ≠ 整篇成品**：你现在库里那些"我自己写的技术文档"是**整篇成品**（走 kb-organize）。原子笔记是把一篇材料**炼小、炼到能互相连接**的产物（走 kb-digest），灵魂是第三段「我的疑问与发散」——结论会过时，但你当时怎么想的，是别人抄不走的复利资产。

## 组成

| 名称 | 类型 | 一句话作用 |
|------|------|-----------|
| `kb-digest` | skill | 给链接/材料 → 先讲解总结（不入库）→ 承接你的追问发散 → 消化后炼成**原子笔记**入库 |
| `kb-weave` | skill | 把新笔记**编进知识网**：找亲戚、牵双向链接、上 MOC、必要时回头修旧账（复利引擎）|
| `kb-ask` | skill | 翻**你自己**的第二大脑作答，溯源到具体笔记，不做 RAG |
| `kb-organize` | skill | 把**已写好的整篇成品**分类归档，只提交本次文件后推送 |
| `kb-librarian` | agent | 整库重活：巡检死链、补全双链、合并重复、批量更新 MOC |

底层脚本（插件根 `scripts/`，各技能复用）：

- `resolve_kb.sh` / `set_kb.sh`：解析 / 设置知识库位置（下方详述）。
- `kb_search.sh`：结构化检索（非 RAG）。
- `commit_kb.sh`：只暂存本次文件的安全提交，绝不 `git add -A`。

```
/plugin marketplace add liusiyangiOS/second-brain-plugin

```

## 首次使用：设置知识库位置（只需一次）

插件**不知道**、也**不该知道**你的知识库在哪——这是它能公开发布、又不泄露任何人本地路径的关键。第一次触发任一技能时，它会解析知识库位置：

- 如果没配置过，技能会**问你一次**："你的知识库根目录绝对路径是？"
- 你回答后，它记录到 `${CLAUDE_PLUGIN_DATA}/kb_path`（或 `~/.config/second-brain-plugin/kb_path`），**之后永远不再问**。

也可以手动设置：

```bash
bash <plugin>/scripts/set_kb.sh /你的/知识库/绝对路径
```

或用环境变量临时覆盖（优先级最高，方便临时切到另一个库）：

```bash
export SECOND_BRAIN_KB_PATH=/另一个/知识库
```

> 你的知识库目录**应当本身是一个 git 仓库**（`git init` 并配好远端），这样 `commit_kb.sh` 才能提交推送。

## 四条典型使用流程

### 流程 1：边学边沉淀（kb-digest，最日常）

```
你：帮我看看这个链接讲的啥 <URL>
→ 我抓取正文，给你「一句话定性 + 结构化摘要」，然后停下
你：（追问）这里的 X 为什么要这样？
→ 我解释，并把这轮问答记进草稿
你：懂了，入库
→ 我在 notes/ 建一条原子笔记（含你的疑问与发散）
→ 自动接 kb-weave 编进知识网
→ commit_kb.sh 只提交这几个文件并推送
```

### 流程 2：整篇成品直接存档（kb-organize）

```
你：把刚才这份方案整理到知识库
→ 我判断分类（如 方案设计/），一句话告诉你放哪
→ 写成带 frontmatter 的 md，commit_kb.sh 提交推送
```

**流程 1 vs 2 怎么选**：还想学明白、要留思考 → kb-digest；已经懂了、只想存档 → kb-organize。

### 流程 3：问自己的第二大脑（kb-ask）

```
你：问下我的知识库，我之前怎么理解「子代理编排」的？
→ 我检索 → 先读领域 MOC → 再读相关原子笔记
→ 基于你的笔记作答，并标注「（见 notes/xxx）」
→ 库里没有就如实说没有，不拿公网内容冒充
```

### 流程 4：定期给知识库做体检（kb-librarian）

```
你：派 kb-librarian 给知识库做次全面体检
→ 先出巡检报告（死链/单向链接/重复条目/MOC 缺口）
→ 你确认后它按主题分批修复并提交
```

## 架构：为什么插件公开、知识库私有

这是刻意的、也是正确的划分：

- **插件（本仓库）= 工具**，公开：不含任何个人数据和本地路径，谁都能装来用。
- **知识库 = 你的数据**，私有：放在你自己的私有仓库里，绝不进本插件。

就像 **VS Code 开源、你的代码私有**。两者通过"你首次设置的知识库路径"解耦——插件在运行时才向你要位置，自己一无所知。所以：

- 换新机器：装插件（公开）+ clone 你的私有知识库，各走各的，互不绑定。
- 你把插件公开发布：别人 clone 下来设置自己的知识库路径就能用，看不到你任何本地信息。

## 知识库结构约定

```
你的知识库/                （本身是一个 git 仓库）
├── _MOC/            # 领域地图（知识网枢纽，随新笔记不断生长）
│   ├── index.md     # 根地图 → 各领域
│   └── <领域>.md
├── notes/           # 原子笔记：一条一个概念，frontmatter 双链
├── 技术分享/ 临时想法/ …   # 整篇成品 / 归档分类目录
└── README.md        # 约定唯一事实源（原子笔记格式、目录规则）
```

`README.md` 是所有技能落盘前必读的**唯一事实源**——想调整笔记格式或分类，改它即可，技能会跟随。
